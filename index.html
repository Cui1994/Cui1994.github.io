直播通Bug修复记录

发表于 2017-01-15 |

此文章用来记录直播通上线后遇到的各种Bug和解决方案，不定期更新。

2017.1.15

网络搜索模块龙珠搜索结果未显示

  • 排查：
    app.Searh中search_longzhu函数出现问题，如果搜索主播未上线就不会出现主播搜索结果。

  • 解决方案：
    因为抓包网址返回的json数据中如果主播上线总是排在第一位的，因此修改函数直接将所抓数据的第一个默认为主播返回搜索结果，当然这样也有问题，就是不管搜索的是不是龙珠主播总会出现一个龙珠搜索结果，不过应该不影响使用。

主播状态验证模块熊猫主播的状态出错

  • 排查：
    查看LiveChecker的日志记录发现访问熊猫网址时出现网络错误max retries exceeded with url

  • 解决方案：
    在网上发现了错误原因，是链接数量太多没有关闭。直接借用其解决方案，增加最大重连次数，并设置http connection的keep-alive状态为关闭。在LiveChecker中加入如下代码：

目前还未发现问题。

Hot页面出现同名主播

  • 排查
    出现同名的原因：如果在网络搜索结果中出现数据库中已有的主播并添加关注时，main.view中的add_new函数会创建一个一模一样的主播，因此出现同名状态。

  • 解决方案
    这个问题比较简单，只要在add_new函数中添加两行判断函数即可，如果数据库中可以搜到name和tv都相同的主播，则不会新建主播对象，而是对数据库中已有的主播对象执行add_user方法即可。

celery是什么

关于celery的知识全部在网上了解，星星点点，但还是有所收获。celery是一个典型的异步任务系统，在应用上下文之外执行任务，将消耗资源的东西通通交给celery来做，可以让主机迅速向应客户端的请求。
celery有三个核心组件：

  • 客户端：在flask系统中和flask一起运行
  • workers：就是传说中的second terminal，用来执行异步任务，可以有多个
  • 消息代理：用来进行celery的通信，一般用redis。

为什么用redis？因为其实时性强，一般用作数据频繁插入，更新或者删除的任务中，以减少对数据库的操作

redis

发表于 2017-01-09 |

redis是具有超高读写速度的key-value数据库系统，同一般的数据库相比，其优势在于是完全储存在内存中的数据，磁盘只用做持久性储存。此外，它的操作都是具有原子性的，已经广泛用在消息队列，网页计数，程序会话等诸多web应用中。昨天看了redis的入门教程，现在进行总结。

1.数据类型

redis拥有比一般数据库更为多样和完整的数据类型

  1. String
    字符串是一个字符序列，具有二进制安全，redis中通过set和get命令读写字符串。

  2. list
    列表通过插入顺序排序，可以选择插入到列表的头部或者尾部

  3. hashes
    键值对集合，表示字符串和字符串之间的映射。

  4. 无序集合
    set中没有重复对象，时间复杂度为O(1)

  5. 有序集合
    增加了一个排序变量，可以允许重复，查询时按排序变量顺序返回。

2.时间相关

redis可以设置key的生存时间。

3.安全相关

为redis设置密码有两种方式，可以修改redis.conf配置文件中的requirepass参数为密码，此外，还可以在进入redis的状态下输入一下命令：

4.事务处理

redis可以用事务上下文MULTI进行事务处理，需要说明的是，事务处理也是有原子性的，即要么都执行，要么都不执行。

5.持久化存储

redis的持久化存储有两种方式。

  • 快照(snapshotting) 直接将数据写入文件
  • aof（appendfsync） 将读写操作写入文件
    二者都可以设置参数，在什么时候进行写入，此处不进行赘述。

6.发布订阅

redis的订阅和发布实现了通讯系统。发布人发布消息，订阅者接受消息。redis一个客户端可以订阅任意数量的通道。
